指定ファイル（またはページ）のコード品質を評価し、点数をつけてリファクタリングする。

## STEP 1: ファイル単位の評価

対象ファイルを読み、各観点を10点満点で採点する。

| 観点 | 見るポイント |
|---|---|
| 可読性 | 変数名・関数名のわかりやすさ、ネストの深さ、1関数の長さ |
| 構造・分割 | 1ファイルの責務が適切か、コンポーネント分割すべき箇所 |
| 重複 | コピペコード、同じロジックの繰り返し、共通化できる処理 |
| データの扱い | props/stateの整理、ダミーデータとの結合度、マジックナンバー |
| Tailwindの使い方 | クラスの冗長さ、一貫性、レスポンシブ指定の統一 |

出力例:
```
[ファイル] EventDetailPage.jsx
  可読性:         6/10 — 1関数300行、条件分岐が深い
  構造・分割:      4/10 — ヘッダー・本文・サイドバーが全部1コンポーネント
  重複:           5/10 — カード表示ロジックが3箇所にコピペ
  データの扱い:    7/10 — propsは整理されている
  Tailwind:       6/10 — レスポンシブ指定がmd/lgで不統一
  小計: 28/50
```

## STEP 2: プロジェクト横断の評価

ファイル単体ではなく、プロジェクト全体から見た観点で採点する。

| 観点 | 見るポイント |
|---|---|
| 命名の統一 | ファイル名・コンポーネント名・変数名の規則が揃っているか |
| 共通化 | 他ファイルと同じパターンの処理が共通コンポーネント/関数に抽出されているか |
| ファイル構成 | 適切なディレクトリに配置されているか、肥大化したファイルがないか |
| import整理 | 未使用import、循環参照、import順の統一 |
| 一貫性 | 同じことを同じやり方でやっているか（map/filter の書き方、イベントハンドラの命名等） |

出力例:
```
[プロジェクト横断] EventDetailPage.jsx
  命名の統一:    8/10 — OK
  共通化:       4/10 — OrganizerCardが3ページで別々に定義
  ファイル構成:  7/10 — OK
  import整理:   5/10 — 未使用importが3件
  一貫性:       6/10 — onClickとhandleClickが混在
  小計: 30/50
```

## STEP 3: 総合スコアと改善実行

```
━━━━━━━━━━━━━━━━━━━━━━━
総合: ファイル単位 XX/50 + プロジェクト横断 XX/50 = XX/100
━━━━━━━━━━━━━━━━━━━━━━━
```

1. スコアが最も低い項目を3つピックアップ
2. それぞれ具体的な改善をコード付きで提案
3. ユーザーの確認後、修正を実行
4. 修正後に再採点して差分を見せる

## ルール
- 機能を変えない（見た目・動作が変わるリファクタは事前に確認）
- 一度に大きく変えず、1項目ずつ改善→再採点
- 新ファイル作成は最小限に

## 手順
1. 対象ファイルのコードを読む（引数でファイル名指定、なければ前回最低スコアまたは未評価）
2. STEP 1 → STEP 2 → STEP 3 の順に実行
3. 改善→再採点を繰り返す

## ループで使う場合
`/loop 10m /refactor-review` でファイルを順番にレビュー→改善を繰り返す。
