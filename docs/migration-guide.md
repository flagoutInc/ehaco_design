# ehaco! フロントエンド リニューアル計画書

## このドキュメントについて

ehaco!（getit-seminar）の画面デザインを新しくするための計画書です。
ehaco_design（デザインモック）をベースに、本番の画面を全面リニューアルします。

---

## 前提条件

- フロントエンドのみ変更。バックエンド（API・DB・AWS）は触らない
- バックエンドの開発・保守はベトナムの委託先が担当
- 新機能に必要なAPI追加はベトナム側に依頼する
- フロントエンドの作業は自分 + Claude Code で行う

---

## リニューアルの全体像

3段階に分けて進めます。

### Phase 0: モック準備（本番移行の前にやること）

ehaco_design（デザインモック）の未作成ページの追加と、コードの整理を行います。
本番移行時にそのまま使える品質に仕上げます。

### Phase 1: 見た目のリニューアル（自分だけで完結）

既存のAPIをそのまま使い、画面の見た目だけを新デザインに変更します。
データの取得・保存の仕組みは何も変わりません。見た目だけが変わります。

### Phase 2: 新機能の追加（ベトナム側との連携が必要）

新しいAPIが必要な機能を追加します。
API仕様書を作成してベトナム側に依頼し、APIができたらフロントを繋ぎます。

---

## Phase 0: モック準備

Phase 1（本番移行）の前に、ehaco_designを完成・整理します。

### 0-1. 管理者画面のモック作成（1〜2日）

ehaco_designに管理者画面がまだありません。以下のページを追加します。

| 画面 | 内容 |
|------|------|
| 管理者ログイン | メールアドレス・パスワードのフォーム |
| ダッシュボード | 管理者トップページ（ユーザー数・主催者数等の概要） |
| ユーザー管理 | ユーザー一覧テーブル・検索・詳細・編集 |
| 主催者承認 | 主催者一覧・ステータス管理（承認/却下） |
| メールキャンペーン | 一覧・作成・編集・送信（リッチテキストエディタ） |
| タグ管理 | タグ・タググループのCRUD |

### 0-2. リファクタリング（1〜2日）

本番移行時にそのまま使えるよう、コードを整理します。

| 作業 | 内容 |
|------|------|
| コンポーネント整理 | 共通コンポーネントの分離・命名統一 |
| ファイル構成 | ページ・コンポーネントのフォルダ構成を本番に合わせる |
| 重複コード削除 | 似たようなUIパターンの共通化 |
| 型定義の準備 | TypeScript移行しやすいようpropsの整理 |
| ダミーデータ整理 | dummy.js / orgDummy.js の構造をAPI仕様に近づける |

### Phase 0 合計: 2〜4日

## Phase 1: 見た目のリニューアル

### やること

- 現在のMUI（Material UI）ベースの画面を、Tailwind CSSベースの新デザインに置き換える
- 呼ぶAPIは全く同じ。データの流れは変わらない
- 全ページ・全画面が対象

### やらないこと

- バックエンド（API・DB）の変更
- 新機能の追加
- データ構造の変更

### 作業内容

| 作業 | 期間 |
|------|------|
| 基盤構築（プロジェクト初期設定・認証接続・共通部品） | 1〜2日 |
| 参加者向け画面（検索・イベント詳細・申込・ログイン等） | 2〜3日 |
| マイページ（ダッシュボード・申込済み・お気に入り・フォロー等） | 1〜2日 |
| 主催者管理画面（イベント管理・申込者管理・アンケート等） | 3〜4日 |
| 管理者画面（ユーザー管理・主催者承認・メール配信等） | 1〜2日 |
| 動作確認・バグ修正 | 3〜5日 |
| **合計** | **2〜3週間** |

### 作業の進め方

1. `dev` ブランチから `redesign/frontend-tailwind` ブランチを作成
2. リポジトリ内に `frontend-redesign/` フォルダを作成（既存の `frontend/` はそのまま残す）
3. `frontend-redesign/` にehaco_designベースの新画面を構築
4. 全画面の動作確認後、`frontend/` と差し替え
5. プルリクエストを出してマージ → デプロイ

### フォルダ構成（作業中）

```
getit-seminar/
├── backend/                ← 変更なし
├── frontend/               ← 現在の画面（そのまま残す）
├── frontend-redesign/      ← 新しい画面（ここで開発）
└── docker/                 ← 変更なし
```

### 切り替え手順

```
1. frontend/ を frontend-legacy/ にリネーム（バックアップ）
2. frontend-redesign/ を frontend/ にリネーム
3. 動作確認
4. 問題なければ frontend-legacy/ を削除
5. デプロイ
```

問題が起きた場合はリネームを元に戻すだけで即座に復旧できます。

## Phase 2: 新機能の追加

### 概要

ehaco_designに含まれる新機能のうち、現在のバックエンドAPIにない機能を追加します。
API仕様書を作成してベトナム側に依頼し、APIが完成したらフロントエンドを接続します。

### 必要なAPI一覧

| 機能 | 内容 | API仕様書作成 | ベトナム側開発 |
|------|------|:---:|:---:|
| 通知システム | 通知一覧・詳細・既読管理・通知設定 | 1日 | 3〜5日 |
| マイページ統計 | 申込数・お気に入り数・フォロー数の集計 | 半日 | 2〜3日 |
| 主催者ダッシュボード統計 | KPI・申込推移グラフ・業種分布 | 半日 | 2〜3日 |
| 主催者一覧 | 公開主催者の検索・一覧取得 | 半日 | 1日 |
| アンケート結果集計 | 回答数・割合・CSV出力 | 半日 | 1〜2日 |
| 主催者→参加者メッセージ | イベント参加者への一斉送信 | 半日 | 2〜3日 |
| イベント詳細Markdown化 | 入力・保存・表示のMarkdown対応 | 半日 | 2〜3日 + データ移行 |

### スケジュール

| 作業 | 期間 | 担当 |
|------|------|------|
| API仕様書作成 | 3〜4日 | 自分（Claude Codeで作成） |
| API開発 | 2〜3週間 | ベトナム側 |
| フロントエンド接続 | 1週間 | 自分 |

Phase 1の完了と同時にAPI仕様書を渡せば、並行して進められます。

## 全体スケジュール

```
Week 1     Phase 0: モック準備（管理者画面作成 + リファクタリング）
Week 1-3   Phase 1: 見た目リニューアル（自分）
Week 3     Phase 2: API仕様書作成・依頼（自分）
Week 4-6   Phase 2: API開発待ち（ベトナム側）
Week 6-7   Phase 2: フロント接続（自分）

合計: 約1.5〜2ヶ月で全て完了
```

Phase 0 と Phase 1 は一部並行して進められます。
Phase 1が終わった時点で、見た目のリニューアルはリリース可能です。
新機能はAPIが揃い次第、追加でリリースします。

## リスクと対策

| リスク | 対策 |
|--------|------|
| 切り替え時に画面が壊れる | バックアップ（frontend-legacy/）から即復旧可能 |
| APIのレスポンス形式が想定と違う | 動作確認期間（3〜5日）で検出・修正 |
| リニューアル中にdevで別の修正が入る | ブランチを分けているので干渉しない |
| 既存のイベントデータ（HTML）のMarkdown変換が崩れる | 新規カラムに変換。元データは残す |
| ベトナム側のAPI開発が遅れる | Phase 1（見た目リニューアル）は先にリリース可能 |

| 項目 | 影響 |

| アカウント情報・申込履歴 | 全て維持 |

| メール | 変わらない |
| ダウンタイム | 切り替え時に数分のメンテナンス |

## ベトナム側に確認が必要なこと

### リニューアル作業について

- [ ] `frontend-redesign/` フォルダをリポジトリに追加して開発を進める方針で問題ないか
- [ ] `redesign/frontend-tailwind` ブランチでの作業に問題ないか
- [ ] 完成後に `frontend/` を差し替えてデプロイする手順に問題ないか
- [ ] デプロイのCI/CD設定に変更が必要か（フォルダ名変更の影響）

### 新機能のAPI開発について

- [ ] Phase 2のAPI仕様書を渡した場合、開発対応は可能か
- [ ] 対応可能な場合、スケジュール感はどのくらいか
- [ ] API仕様書のフォーマットに指定はあるか（OpenAPI等）

### イベント詳細のMarkdown化について

- [ ] 現在のイベント詳細文のデータ形式は何か（HTML? リッチテキスト?）
- [ ] 既存イベントデータは何件あるか
- [ ] 新しいカラム（description_md）を追加してMarkdown版を保存する方針で問題ないか
- [ ] 既存データのMarkdown変換スクリプトの実行はベトナム側で対応可能か

### その他

- [ ] フロントエンドのパッケージ変更（MUI削除 → Tailwind CSS導入）に制約はあるか
- [ ] 本番デプロイのタイミングに制約はあるか（リリース凍結期間等）
